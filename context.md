## Architecture

### Current architecture

The bridge is TypeScript/Node and Telegram-driven:

- `src/config.ts` loads `.env`, validates roots, profiles, Pi binary/args, and workspace isolation.
- `src/config-wizard.ts` inspects and can initialize project-local assets, but current active project lacks `.agents` and `.mcp`.
- `src/agent-router.ts` manages profile runtimes/workspaces (not deeply inspected here).
- `src/lab.ts` creates lab prompts and executes clone-profile sessions.
- `src/lab-reports.ts` stores report records as JSONL in `${AGENT_WORKSPACE_ROOT}/reports/lab-runs.jsonl`.
- `src/index.ts` wires Telegram commands and uses the default profile as the orchestrator for triage and Engram sync prompts.

The lab flow is already conceptually a small state machine (`pending` -> `triaged` -> decision -> Engram approval/save), but JSONL makes querying, deduping, and cross-report status tracking awkward.

### Recommended simple SQLite schema

Use one DB file under the same workspace root, for example:

```text
${AGENT_WORKSPACE_ROOT}/reports/lab.db
```

Keep JSONL export optional for human/debug compatibility.

Suggested minimal schema:

```sql
PRAGMA foreign_keys = ON;

CREATE TABLE lab_runs (
  id TEXT PRIMARY KEY,
  project_id TEXT NOT NULL,
  project_path TEXT NOT NULL,
  agent_id TEXT NOT NULL,
  agent_label TEXT NOT NULL,
  workspace TEXT NOT NULL,
  duration_label TEXT NOT NULL,
  duration_ms INTEGER NOT NULL,
  status TEXT NOT NULL CHECK (status IN ('running','completed','failed','timeout','skipped')),
  summary TEXT NOT NULL,
  raw_output TEXT,
  error TEXT,
  started_at TEXT NOT NULL,
  finished_at TEXT NOT NULL,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE bug_findings (
  id TEXT PRIMARY KEY,
  run_id TEXT NOT NULL REFERENCES lab_runs(id) ON DELETE CASCADE,
  project_id TEXT NOT NULL,
  title TEXT NOT NULL,
  description TEXT NOT NULL,
  severity TEXT NOT NULL CHECK (severity IN ('critical','high','medium','low','info')),
  confidence TEXT NOT NULL CHECK (confidence IN ('high','medium','low')),
  status TEXT NOT NULL DEFAULT 'new' CHECK (status IN ('new','triaged','accepted','deferred','ignored','fixed','duplicate')),
  evidence TEXT,
  suspected_cause TEXT,
  affected_files TEXT, -- JSON array of paths
  dedupe_key TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE UNIQUE INDEX bug_findings_dedupe_idx
  ON bug_findings(project_id, dedupe_key)
  WHERE dedupe_key IS NOT NULL;

CREATE TABLE finding_status_events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  finding_id TEXT NOT NULL REFERENCES bug_findings(id) ON DELETE CASCADE,
  old_status TEXT,
  new_status TEXT NOT NULL,
  actor TEXT NOT NULL, -- lab_agent | orchestrator | user | sync
  note TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE proposals (
  id TEXT PRIMARY KEY,
  finding_id TEXT NOT NULL REFERENCES bug_findings(id) ON DELETE CASCADE,
  proposal_type TEXT NOT NULL CHECK (proposal_type IN ('fix','test','investigation','docs','memory')),
  summary TEXT NOT NULL,
  details TEXT,
  priority INTEGER NOT NULL DEFAULT 3,
  status TEXT NOT NULL DEFAULT 'proposed' CHECK (status IN ('proposed','approved','rejected','implemented','superseded')),
  created_by_agent_id TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  decided_at TEXT
);

CREATE TABLE lab_tasks (
  id TEXT PRIMARY KEY,
  finding_id TEXT REFERENCES bug_findings(id) ON DELETE SET NULL,
  proposal_id TEXT REFERENCES proposals(id) ON DELETE SET NULL,
  project_id TEXT NOT NULL,
  task_type TEXT NOT NULL CHECK (task_type IN ('reproduce','verify','fix','review','sync_engram')),
  title TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'pending' CHECK (status IN ('pending','running','blocked','done','cancelled')),
  assigned_agent_id TEXT,
  command_budget INTEGER,
  notes TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE skill_index (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  path TEXT NOT NULL UNIQUE,
  source TEXT NOT NULL CHECK (source IN ('project','source','user','global')),
  description TEXT,
  priority INTEGER NOT NULL DEFAULT 100,
  fingerprint TEXT,
  indexed_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE finding_skills (
  finding_id TEXT NOT NULL REFERENCES bug_findings(id) ON DELETE CASCADE,
  skill_id TEXT NOT NULL REFERENCES skill_index(id) ON DELETE CASCADE,
  reason TEXT,
  PRIMARY KEY (finding_id, skill_id)
);
```

Practical notes:

- Store raw lab output once in `lab_runs.raw_output`; store structured extracted problems in `bug_findings`.
- Use `dedupe_key` such as `project_id + normalized title + affected file + failing command` to collapse repeated lab discoveries.
- Use `finding_status_events` for auditability instead of overwriting history only.
- Store `affected_files` as JSON text initially; only normalize later if needed.
- Keep `proposals` separate from findings so multiple agents can suggest competing fixes/tests.
- Keep `lab_tasks` as the actionable queue for orchestrator/user decisions.
- `skill_index` should include project-local skills first, then imported source/user/global indexes as secondary context.

## How lab prompts should consume the DB and skill index

Recommended prompt/data flow:

1. Before launching a lab run, bridge queries the DB for:
   - open high/critical findings for the same project;
   - recently duplicated findings/dedupe keys;
   - pending lab tasks assigned to this agent/profile or unassigned;
   - relevant skills from `skill_index`, ordered by `source='project'` first and then by priority.
2. The lab prompt should receive a compact, read-only context block, not the whole DB:

```text
## Known open findings
- <id> [severity/status/confidence] <title> affected: <paths> last evidence: <short>

## Assigned lab tasks
- <id> <task_type>: <title> command_budget=<n>

## Skills to inspect before work
1. <path-to-project-local-skill/SKILL.md> - reason
2. <path-to-source-or-user-skill/SKILL.md> - reason
```

3. Prompt should require structured output after the human-readable summary:

```json
{
  "findings": [
    {
      "title": "...",
      "severity": "high|medium|low|info",
      "confidence": "high|medium|low",
      "evidence": "...",
      "suspected_cause": "...",
      "affected_files": ["..."],
      "dedupe_key": "...",
      "proposals": [{ "type": "fix|test|investigation|docs", "summary": "..." }]
    }
  ],
  "task_updates": [
    { "task_id": "...", "status": "done|blocked", "note": "..." }
  ],
  "commands": [
    { "command": "...", "outcome": "pass|fail|error", "evidence": "..." }
  ]
}
```

4. Bridge parses the structured block into `bug_findings`, `proposals`, `lab_tasks`, and status events.
5. The triage prompt should consume the DB row(s), not raw JSONL only, and should update statuses/proposals rather than just writing a text `triageRaw` blob.
6. `/reports` should list findings grouped by status/severity, with drilldowns to run evidence.
7. `/report <id> work|defer|ignore|save` should target a `bug_findings.id` or `lab_runs.id` explicitly; current `report` wording is lab-run centric and will become ambiguous once findings are first-class.
8. `/syncreports` should save accepted/deferred/saved finding summaries, not raw lab runs.

Skill index consumption order:

1. Prefer `./.agents/skills/**/SKILL.md` in the active project.
2. Then consult `./.atl/skill-registry.md` for known user/global skills.
3. Optionally index the inspected source skills from `Sistema_de_mantencion/.agents/skills` as `source='source'`, but do not auto-copy them into this project without user approval.
4. Special-case `buenas-practicas-bd` only if DB/schema work is involved; otherwise avoid forcing it into every lab prompt.
5. Keep prompt skill lists short: pass exact paths for 1-3 relevant skills, not the whole inventory.

## Risks

- **Secret exposure:** `.env` contains Telegram credentials; diagnostics and reports must never print `.env` wholesale. This scout intentionally omitted secret values.
- **Conflicting Pi args:** `.env` has `--no-skill-registry --no-lens`, while this feature wants DB+skill index consumption. Decide whether bridge-level indexing replaces Pi skill registry or whether flags should change.
- **No project-local `.agents` or `.mcp`:** current active project cannot satisfy lab prompt project-local skill/MCP checks until assets are initialized or copied.
- **Source skills are from another project:** importing `Sistema_de_mantencion/.agents` into `pi-telegram-bridge` could mix domain-specific RCM skills with bridge/runtime skills. Treat as source inventory/proposals, not automatic install.
- **JSONL to SQLite migration:** existing reports may be in `<bridge-agents-root>/reports/lab-runs.jsonl`; migration should be read-only/import-safe and idempotent.
- **Concurrent lab writes:** labs run in parallel. SQLite needs serialized writes or short transactions with busy timeout. JSONL append was simple but less queryable.
- **Ambiguous IDs:** current `/report <id>` means lab run id. With findings/tasks, command syntax should distinguish run vs finding or accept prefixes.
- **Engram availability:** this subagent tool namespace exposes no Engram/memory tools, so the requested Engram save for significant discoveries could not be performed.
- **MCP safety:** `.mcp/config.json` is intentionally missing/disabled by default; do not execute newly discovered MCP servers automatically.
